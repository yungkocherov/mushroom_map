# Обзор проекта mushroom-map — простыми словами

Это обзор для человека, который только-только открывает репозиторий и хочет
понять: что это за проект, из каких кусков он состоит, что каждый кусок делает
и зачем он нужен.

Если тебе нужна детальная техничка для AI-ассистента — читай [CONTEXT.md](../CONTEXT.md).
Если архитектуру БД и API — [architecture.md](architecture.md).
Если планы по точности лесного слоя — [copernicus_migration.md](copernicus_migration.md).

---

## 1. Что это вообще такое

**Mushroom-map** — это веб-сервис «грибная карта Ленинградской области».

Идея простая: открываешь сайт, видишь карту Ленобласти, где леса закрашены
по породе деревьев (сосняки — одним цветом, ельники — другим, березняки —
третьим и т.д.). Тыкаешь в лес — получаешь попап с двумя списками грибов:

1. **Теоретический список** — «что тут *должно* расти» по справочнику.
   Например, в сосняке белые, маслята, лисички; в березняке — подберёзовики.
2. **Эмпирический список** — «что тут *реально* находили люди» по постам из
   ВК-групп типа «Грибники Ленобласти». Если за 5 лет в этой клетке карты
   пять раз находили рыжики — значит рыжики тут есть, а не просто «по теории».

Архитектура заложена так, чтобы проект легко масштабировался:

- легко добавлять новые регионы (не только Ленобласть),
- легко подключать точные данные о лесах (Copernicus HRL, 10-метровые
  спутниковые данные о породах деревьев),
- легко добавлять новые источники наблюдений (iNaturalist, GBIF, UGC).

## 2. Как всё работает (общий поток данных)

Представь конвейер. На входе — сырые данные из интернета, на выходе — попап
на карте. Вот как данные превращаются:

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   OSM (леса)     │ →→→ │ forest_polygon   │ →→→ │  PMTiles (карта) │
│ Overpass API     │     │ в PostGIS        │     │  в браузере      │
└──────────────────┘     └──────────────────┘     └──────────────────┘
                              ↓
                         лесной слой
                              ↓
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ Справочник видов │ →→→ │ species +        │ →→→ │ попап «теория»:  │
│ (yaml-файл)      │     │ species_forest_  │     │ что должно тут   │
│                  │     │ affinity         │     │ расти            │
└──────────────────┘     └──────────────────┘     └──────────────────┘

┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  ВК-посты        │ →→→ │ распознавание    │ →→→ │  observation     │
│ (wall.get)       │     │ фото через Gemma │     │ в PostGIS        │
└──────────────────┘     └──────────────────┘     └──────────────────┘
                                                         ↓
┌──────────────────┐     ┌──────────────────┐            │
│  OSM (топонимы)  │ →→→ │ gazetteer_entry  │            │
│ места, озёра     │     │ + admin_area     │            │
└──────────────────┘     └──────────────────┘            │
                              ↓                          ↓
                         NER (Natasha) + match → point + h3_cell
                              ↓
                         попап «эмпирика»:
                         что тут реально находили
```

Каждая стрелка — это отдельный пайплайн-скрипт, который можно запустить
независимо от остальных. База данных (PostGIS) — клей, который всё держит
вместе.

## 3. Четыре слоя данных

Проект делится на четыре логических слоя. Каждый слой — отдельная задача.

### 3.1. Лесной слой: «где какой лес»

**Что это:** 47 000 полигонов Ленобласти, у каждого есть порода дерева
(сосна, ель, берёза, смешанный и т.д.) или `unknown`, если порода неизвестна.

**Откуда берётся:** [pipelines/ingest_forest.py](../pipelines/ingest_forest.py)
дёргает Overpass API (это фронтенд к OpenStreetMap), выкачивает все полигоны
с тегами `landuse=forest` и `natural=wood` в bbox Ленобласти, парсит теги
`leaf_type` / `wood`, нормализует в наш единый словарь пород и кладёт в
таблицу `forest_polygon`.

**Куда кладётся:** таблица [forest_polygon](../db/migrations/004_forest.sql).

**Как попадает на карту:** [pipelines/build_tiles.py](../pipelines/build_tiles.py)
генерирует из полигонов **PMTiles** — это один большой файл (~54 МБ) с
векторными тайлами. Браузер скачивает кусочки этого файла по мере зума и
рисует полигоны прямо на клиенте (через MapLibre GL). Это быстро и бесплатно —
не нужен отдельный тайловый сервер.

**Проблема:** 88% полигонов у нас — `unknown` (серые). В российском OSM
почти никто не размечает породы деревьев. Решений два:
- Быстрое: натравить эвристику на названия (если полигон называется «Сосновый
  бор» — значит сосна). Ограниченно поможет.
- Правильное: перейти на **Copernicus HRL Tree Species** — это европейский
  спутниковый продукт с разрешением 10 м, где для каждого пикселя уже
  определена порода. План перехода — в [copernicus_migration.md](copernicus_migration.md).

### 3.2. Справочник грибов: «что где должно расти»

**Что это:** 24 вида грибов с названиями (русское + латинское), сезонностью,
съедобностью и привязкой к типам леса (афинность 0..1).

**Где хранится:** [db/seeds/species_registry.yaml](../db/seeds/species_registry.yaml) —
человекочитаемый файл, единственный источник истины. Оттуда генерируется
SQL-сид [species_registry.sql](../db/seeds/species_registry.sql), который
запускается после миграций.

**Зачем:** когда пользователь тыкает в сосновый лес, API делает запрос «какие
виды имеют высокую афинность к `pine`?» — и получает список белых, маслят,
лисичек, горькушек и т.д. Это **теоретический** список из попапа.

**Таблицы:**
- `species` — сам гриб (`slug`, `name_ru`, `name_lat`, `edibility`, `season_months`).
- `species_forest_affinity` — связь вид↔тип_леса со значением афинности.

### 3.3. Наблюдения из ВК: «что тут реально находили»

Это самый сложный слой, потому что он работает с мусорными текстами и
фотографиями, а не структурированными данными.

Пайплайн [pipelines/ingest_vk.py](../pipelines/ingest_vk.py) состоит из
**четырёх стадий**, каждая пишет чекпоинт в `data/vk/{group}/` чтобы можно
было безопасно прервать и продолжить:

1. **`collect`** — качает все посты из ВК-группы через `wall.get` (до 8 лет
   назад). Сохраняет `raw_posts.json`: id, дата публикации, текст, список
   URL фотографий.
2. **`dates`** — из текста поста извлекает **дату похода**. Часто люди
   постят «в субботу 5-го октября» — реальная дата сбора на день-два раньше
   даты поста. Пайплайн пытается это распарсить regex'ом, при неуверенности
   может позвать LLM (Claude API).
3. **`photos`** — каждую фотографию показывает локальной мультимодальной
   модели **Gemma 3 12B** (запущенной в LM Studio) и просит определить,
   что за гриб на фото. Gemma возвращает одну из 8 групп: `bolete`,
   `chanterelle`, `honey_fungus`, `morel`, `russula`, `lactarius`, `amanita`,
   `parasol`, или `none` / `other`. Это нарочно грубая классификация — фото
   из ВК часто мыльные, без чётких деталей.
4. **`db`** — записывает итог в таблицу `observation`. Каждая
   фото-распознавание → строка с привязкой к виду, дате похода, тексту поста.

Итог: таблица `observation` наполняется данными, но **без координат**.
Координаты добавляет следующий слой.

**Сезонный фильтр (TODO):** сейчас у нас одна группа `bolete` мапится сразу
на несколько видов (белый, подберёзовик, подосиновик). После первого прогона
надо разделить их в промпте Gemma + добавить проверку «сморчки только
в апреле-мае, иначе отбросить как шум распознавания».

### 3.4. Привязка к местам (NER) — *новый слой, сделали вчера*

**Проблема:** у нас есть observation с видом гриба и датой, но без точки на
карте. Где именно собрали? Человек написал «Ездили в субботу в Лемболово,
набрали белых». Нужно понять: «Лемболово» — это посёлок с координатами
(60.24, 30.19) → положить наблюдение туда.

**Решение — два шага:**

1. **Gazetteer** (справочник топонимов). Таблица `gazetteer_entry` содержит
   все посёлки, хутора, озёра, горы, станции в bbox Ленобласти, с
   координатами и типом (`settlement`, `lake`, `station`, `tract`, `river`).
   Собирается пайплайном [pipelines/load_gazetteer.py](../pipelines/load_gazetteer.py)
   через тот же Overpass API — запросы `place=village|town|hamlet|...`,
   `natural=peak|water`, `railway=station`, `waterway=river|canal`.
   Одновременно качаются и **админ-районы** Ленобласти (boundary=administrative,
   admin_level=6/8) → таблица `admin_area`.

2. **Матчинг поста → топоним.** Пайплайн [pipelines/extract_places.py](../pipelines/extract_places.py)
   для каждого поста:
   - Прогоняет текст через **Natasha NER** (библиотека для русского NER,
     находит LOC-сущности) → получает список упоминаний с нормализованными
     леммами: `["Лемболово", "Орехово"]`.
   - Каждое упоминание ищет в gazetteer через `GazetteerMatcher`
     (файл [services/placenames/src/placenames/geocode.py](../services/placenames/src/placenames/geocode.py)):
     - сначала точное совпадение по нормализованному имени,
     - потом по алиасам (`alt_name`, `old_name` из OSM),
     - потом нечёткое сравнение через `pg_trgm.similarity` (на случай опечаток).
     Дизамбигуация (если совпадений несколько): сначала поселения, потом
     станции, потом урочища; при равном типе — по популярности.
   - Выбирает лучшее совпадение, считает `h3_cell` через библиотеку H3
     (гексагональная сетка Убера, ресолюшн 7 ≈ 5 км² на гекс).
   - Обновляет observation: `point`, `h3_cell`, `placename_raw`,
     `placename_confidence`, в `meta` кладёт тип матча и id gazetteer-записи
     для дебага.

**Почему H3 а не просто точка:** точка выглядит точной, но это обман —
«Лемболово» это целый посёлок + окружающие леса, а не одно место. H3-ячейка
(~5 км²) честно отражает неопределённость. Плюс агрегация «сколько раз в
этой клетке находили белые за октябрь 2023» делается одним GROUP BY.

**Где это всё хранится:**
- [services/placenames/src/placenames/ner.py](../services/placenames/src/placenames/ner.py) —
  обёртка вокруг Natasha.
- [services/placenames/src/placenames/gazetteer.py](../services/placenames/src/placenames/gazetteer.py) —
  нормализация имён, Overpass-запросы, upsert в БД, сборка геометрии
  admin_area из relation-кусков.
- [services/placenames/src/placenames/geocode.py](../services/placenames/src/placenames/geocode.py) —
  `GazetteerMatcher` с тремя стратегиями + стоп-лист («Россия», «СПб», «лес»).

## 5. Основные команды

```bash
# Поднять БД (если Docker Desktop запущен)
docker compose up -d db

# Всё в одном флаконе (db + api + web)
docker compose --profile full up -d

# Миграции (первый раз)
.venv/Scripts/python db/migrate.py

# Загрузить лесные полигоны (первый раз или для обновления OSM)
python pipelines/ingest_forest.py --region lenoblast

# Сгенерировать PMTiles после обновления лесов
python pipelines/build_tiles.py

# Загрузить газеттир (районы + посёлки + озёра) ← НОВОЕ
python pipelines/load_gazetteer.py --region lenoblast

# Собрать ВК-данные (все 4 стадии подряд)
./scripts/ingest_vk.sh

# Привязать собранные посты к местам (NER + матч) ← НОВОЕ
python pipelines/extract_places.py --region lenoblast --group grib_spb
```

## 6. Что работает, а что пока нет

### Работает ✅

- PostGIS со всеми таблицами и 7 миграциями.
- 47 000 лесных полигонов загружены.
- PMTiles генерируется и отдаётся через FastAPI.
- Карта в браузере показывает леса, раскрашенные по породам.
- Клик на лес → попап с названием + теоретический список грибов.
- VK-парсер полностью написан (4 стадии).
- Агрегация наблюдений по регионам через матвью работает.
- Natasha NER + газеттир-матчер написаны и покрыты smoke-тестами.

### Готово, но ещё не запускалось ⏳

- **Газеттир Ленобласти** — код есть, нужен один запуск
  `python pipelines/load_gazetteer.py --region lenoblast` при живой БД.
- **Первый прогон VK-парсера** — нужен действующий `VK_TOKEN` и
  запущенный LM Studio с Gemma для стадии `photos`.
- **extract_places** — будет работать когда будет и газеттир, и ВК-данные.

### Не сделано ❌

- **Улучшение лесного слоя:** 88% полигонов — `unknown`. Нужен Copernicus
  или хотя бы эвристика по именам.
- **Фильтры в UI:** sidebar существует как компонент, но логики фильтрации
  по сезону/виду/съедобности нет.
- **Кликабельная легенда.**
- **Мобильная вёрстка.**
- **Разделение группы `bolete` на отдельные виды** в промпте Gemma —
  отложено до первого реального прогона, чтобы увидеть что она реально
  путает.

## 7. Если что-то пошло не так

- **Порт 5432 занят на Windows** — используй 5434 (прописан в `.env.example`).
- **`localhost` на Windows резолвится в IPv6** — в `vite.config.ts` уже
  прописан `http://127.0.0.1:8000`, не меняй на `localhost`.
- **PMTiles не грузится** — проверь что `TILES_DIR` в `services/api/.env`
  показывает на `../../data/tiles`.
- **Overpass 429/503** — пайплайны сами переключаются между 3 зеркалами
  (`overpass-api.de`, `kumi.systems`, `maps.mail.ru`).
- **Natasha падает с `pkg_resources`** — нужен `setuptools<81` в venv
  (pymorphy2 использует устаревший API, который выпилили в 82).
