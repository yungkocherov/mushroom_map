# Website plan — mushroom-map как часть полноценного сайта

Статус: **Фаза 1 выполнена (2026-04-17)**, Фазы 2–8 в работе.
Документ отражает текущие решения; по мере прохождения фаз отмечаются
✅. Главный вопрос: как превратить карту в полноценный сайт с личным
кабинетом, блогом, справочником видов, методологией — без потери того
что уже сделано.

**Принятые решения** (по итогам обсуждения):
- Framework: **Vite + react-router-dom + vite-ssg** (не Next.js — для
  нашего масштаба избыточно, см. §5).
- Кабинет: **отложен**, не в MVP. Стартуем без auth.
- Блог: **не делаем** в ближайших фазах. Плейсхолдер тоже не нужен
  (негативный сигнал). `app/blog/` добавится когда появится материал.
- Деплой: **Cloudflare Pages + R2 + Oracle Cloud Free Tier** — одобрен.

Это самостоятельный документ. Живая дорожная карта фич — в
[roadmap_content_ideas.md](roadmap_content_ideas.md); архитектура
бэкенда — в [architecture.md](architecture.md). Этот план — надстройка
над ними, оформляет переход от «карта» к «сайт про грибы, в центре
которого карта».

## 1. Цели и не-цели

### Цели
- Карта остаётся **главной ценностью** — первый экран, всегда на
  расстоянии клика.
- Добавить страницы которые **дополняют** карту: справочник видов,
  методология (откуда данные и с какими оговорками), блог/находки,
  «об авторе», личный кабинет с сохранёнными местами.
- Стилистика — **природа и грибы**, но не китч. Ближе к ботаническому
  атласу XIX века или академическому полевому определителю, чем к
  мультяшным иконкам.
- Отдельный фокус — **мобильная версия**: сбор грибов происходит в поле,
  где открывают карту с телефона. Не PWA-обязательно на первом этапе,
  но адаптив обязателен.
- Развёртывание **бесплатно** для старта; переход на платный домен —
  отдельное решение позже.

### Не-цели (осознанные)
- Соцсеть грибников. Чужие посты/лайки/комменты усложняют модерацию и
  правовые риски (сбор в ООПТ). Одиночный «личный дневник» — да, публичная
  лента — нет.
- Identification AI («сфотографируй гриб, узнай что это»). Это отдельный
  продукт, большая ответственность — ошибка = отравление. Ссылаться на
  внешние инструменты (iNaturalist) — ок. Делать своё — нет.
- Монетизация. Просто уважительно к себе и пользователям: без рекламы,
  без трекеров.

## 2. Информационная архитектура (карта сайта)

```
/                      Главная (герой-секция → карта)
├── /map               Полноэкранная карта (текущее приложение)
├── /species           Каталог видов грибов
│   └── /species/[slug]    Страница вида (фото, сезон, тип леса, рецепты)
├── /guide             Гайды и best practices
│   ├── /guide/seasons     Сезоны ЛО: что в каком месяце
│   ├── /guide/safety      Съедобность, двойники, отравления
│   ├── /guide/legal       Законы о сборе (ООПТ, защитные леса)
│   └── /guide/equipment   Что брать с собой: корзина, нож, навигатор
├── /methodology       Как собраны данные (ФГИС ЛК, OSM, species affinity)
│   ├── /methodology/forest-data   Rosleshoz, Copernicus, OSM — что и почему
│   ├── /methodology/species       Справочник: источники, ограничения
│   └── /methodology/observations  VK-парсер, H3-агрегация (когда появится)
├── /blog              Записи: находки сезона, обновления, размышления
│   └── /blog/[slug]
├── /about             Об авторе, контакты, причины проекта
├── /cabinet           Личный кабинет (auth)
│   ├── /cabinet/spots     Сохранённые места с заметками
│   ├── /cabinet/trips     Поездки: когда, что нашёл
│   └── /cabinet/settings  Настройки: предпочтительные виды, напоминания
└── /api/...           Открытые эндпоинты для тех, кто хочет строить своё
```

Принципы:
- **Карта — всегда в шапке**, клик — открывает /map без перезагрузки.
- **URL-sync** карты сохраняется (lat/lon/z/layers), чтобы ссылки в
  блоге работали как «посмотри вот эту полянку».
- **Глубокие ссылки на виды**: `/species/белый-гриб?season=august` —
  сразу фильтр на карте + текст страницы.

## 3. Дизайн

### Настроение

Представь старый кожаный блокнот натуралиста: тёплая бумажная текстура,
ботаническая иллюстрация в две краски, аккуратный академический шрифт,
свободное пространство. Не хипстерский минимализм, не «modern SaaS» —
**полевой определитель**.

Референсы (смотреть и брать идеи, не копировать):
- Field Museum / Royal Botanic Gardens Kew — академический ботанический
  стиль, строгая типографика, пастельные иллюстрации.
- [are.na](https://www.are.na/) — ощущение «медленного, вдумчивого
  контента», без визуального шума.
- Книги издательства «Альпина Нон-фикшн» и серии «Библиотека Абсурдопедии» —
  русскоязычная традиция научпопа, серифный шрифт, drop caps.
- Ernst Haeckel «Kunstformen der Natur» (1904) — иллюстрации-референс
  для декоративных элементов.

### Цветовая палитра

```
Base (светлая тема):
  --paper:       #f6f2e7   фон — слоновая кость, тёплая бумага
  --ink:         #2a2620   основной текст — тёмный графит, не чёрный
  --ink-dim:     #6e6659   вспомогательный текст
  --rule:        #d8d0bd   тонкие линии, границы карточек

Лесные акценты (от существующих FOREST_COLORS):
  --forest:      #2d5a3a   хвойный лес, основной брендовый
  --forest-deep: #1a3a24   hover-состояние, заголовки
  --moss:        #7a9b64   мшистый светлый
  --chanterelle: #d88c1e   тёплый охряно-оранжевый — edible-виды, CTA
  --birch-bark:  #e8e2d1   очень светлый — hover на карточках
  --danger:      #8b2a2a   toxic-виды, приглушённый красный

Dark mode:
  --paper:       #171812   глубокий лесной сумрак
  --ink:         #e4dfd0   тёплая слоновая кость
  --ink-dim:     #8f8979
  --forest:      #6ba878   светлее для контраста на тёмном
  --rule:        #2a2c22
```

Принципы использования:
- Основной фон — **paper**, не белый. Холодный white-on-white делает
  сайт похожим на SaaS. Тёплый бежевый сразу даёт «нечто природное».
- Акценты — **forest**, **chanterelle** (для действий) и очень умеренно.
- Dark mode — не инвертированные цвета, отдельная «ночная лесная»
  палитра. Глубокий зелёно-чёрный, приглушённый свет.

### Типографика

```
Заголовки: Fraunces (serif, academic, variable)  - бесплатно с Google Fonts
Body:      Inter (sans, очень читаемый)           - бесплатно
Mono:      JetBrains Mono                         - для координат и кода
```

Сочетание serif-заголовков + sans-body — классика редакторских сайтов
(New York Times, Atlantic). Fraunces — современная вариация антиквы,
есть вариант со стилизацией «soft» который даёт чуть тёплое ощущение.

**Не** нужно: Montserrat, Roboto (слишком SaaS), Comic Sans (не китч).

### Типографическая шкала
```
Display:   2.5rem / 3rem  (hero)
H1:        2rem
H2:        1.5rem
H3:        1.25rem
Body:      1rem (16px)
Small:     0.875rem
Micro:     0.75rem (captions, metadata)
line-height body: 1.65
line-height headings: 1.2
```

### Иллюстрации и декор

- **Ботанические SVG-иконки** тонкой линией (1.5px stroke) для
  навигации и списков — не эмоджи 🍄 (уже сейчас использую, на финальном
  сайте уберу).
- **Маленькие декоративные плашки** (плодовые тела в две краски, как
  в атласе) на границах секций. Можно нарисовать 8-10 штук, ставить
  случайно.
- **Фоновая текстура** — едва заметное зерно бумаги (SVG-noise или
  PNG-overlay), opacity 0.04–0.06. На мобильном отключать — экономия
  трафика.
- **Никаких стоковых фото** от природы. Либо собственные фото
  грибов/лесов, либо ботанические иллюстрации (public domain).

### Layout-паттерны

- Контент-страницы (блог, гайды, методология) — узкая колонка **640–720px**
  для комфортного чтения. Не full-width. Это книга, не дашборд.
- **/map** — единственная страница full-screen, без рамок.
- **/species/[slug]** — двухколоночная: слева фото/иллюстрация + факты,
  справа длинный текст про экологию, сезон, двойники.
- **/cabinet** — таблицы/списки, но в «книжной» стилистике: не material-карточки,
  а строки с тонкими разделителями.

## 4. Функционал и наполнение

### Справочник видов — главный дополняющий раздел к карте

У нас уже есть `species_registry` с 24 видами в БД. Каждый вид — страница:
- Фото (свои + ссылки на wikimedia).
- Русское и латинское имя, синонимы, edibility-статус с понятной разметкой.
- **Сезон** — мини-график 12 месяцев, подсвечены активные.
- **Типы леса** — какие, в виде цветных плашек из FOREST_COLORS.
- **Сходные виды** (двойники) — критично для безопасности.
- Кнопка **«Показать на карте»** → /map с фильтром по этому виду.
- Раздел кулинария (рецепты, подготовка) — отдельный аккуратный блок.

Каждый вид → автоматически генерируется при билде, SEO-friendly,
ссылки из блога и из попапа карты.

### Блог и полевые заметки

Формат `/blog/2026-08-03-пасмурный-август` — простые markdown-посты с
фронтматтером:

```
---
title: Пасмурный август под Токсово
date: 2026-08-03
place: [60.1234, 30.1234]  # → появится точка на карте в блоке поста
species_found: [белый-гриб, лисичка-обыкновенная]
---
```

Каждый пост может включать мини-карту (тот же MapLibre, изолированный
вид) с точкой находки. Посты индексируются по тегам (места, виды,
сезоны).

Это мощный рычаг: блог даёт «живую» ценность, привязанную к карте.
Пользователь читает «Пасмурный август» → кликает координату → видит
тот же лес на карте → понимает какой это бор по данным ФГИС ЛК.

### Методология

Секция про **доверие**. Текущие каверзы в данных:
- Rosleshoz зачастую не обновлялся с 2015, часть выделов устарели.
- Copernicus до России не дотягивается.
- OSM — только там где люди размечали.
- species-affinity — экспертные, не статистика; честно пишем «gut feeling».
- VK-наблюдения (когда появятся) — шум, NER-ошибки, 5-10% точности
  координат.

Эта честность — конкурентное преимущество. 99% грибных сайтов делают
вид что их данные идеальные. Если мы покажем ограничения — те кто
поймёт, будут доверять больше.

### Личный кабинет (MVP)

Три функции, больше не надо пока:
1. **Сохранённые места** — клик на карте → «сохранить как своё место» →
   название, заметка, фото. Видны только владельцу.
2. **Журнал поездок** — дата, место, что нашёл, заметка. Привязка к
   виду из справочника.
3. **Настройки** — email для пушей «в ЛО наступает сезон белых»
   (опционально), предпочитаемые виды.

Всё остальное (расшаривание мест, соцфункции) — позже если попросят.

### API

Просто задокументировать что уже есть: `/api/forest/at`, `/api/species/search`,
`/tiles/*.pmtiles`. Добавить `/api/species` (список видов для каталога),
`/api/spots/mine` (для кабинета). Это бесплатный бонус для тех кто
захочет собственное что-то построить.

## 5. Технический стек

### Что есть сейчас
- **Frontend**: React 18 + Vite SPA. Одна страница — карта.
- **Backend**: FastAPI + PostGIS + PMTiles.
- **Build**: `npm run build` → `dist/` статика. PMTiles отдаёт FastAPI.

### Что нужно для многостраничного сайта

Выбор — три варианта.

#### Вариант A: Next.js 14+ (App Router) — **рекомендуемый**
Плюсы:
- SSR/SSG для контент-страниц (блог, методология, виды) → идеальный SEO.
- Встроенный роутинг, MDX-плагин для блога, ISR для видов.
- Серверные компоненты (RSC) — меньше JS в клиенте.
- API routes — можно постепенно переносить эндпоинты с FastAPI на Next.js
  (или оставить FastAPI как есть, Next.js проксирует).
- Экосистема (shadcn/ui для UI, next-auth для auth).
- MapLibre изолируется в `app/map/page.tsx` с `'use client'`.

Минусы:
- Миграция — неделя работы (переписать `services/web` под Next-структуру).
- Bundle больше чем у Vite. Для контент-страниц — не критично.

#### Вариант B: Astro + React islands
Плюсы:
- Контент-страницы — zero JS, супер-быстрые.
- Карта живёт в island `<MapView client:only="react">`.
- MDX нативно.

Минусы:
- Для личного кабинета с dynamic-функциями (auth, записи) — Astro
  менее удобен. Придётся подпирать серверлес-функциями.
- Меньше ecosystem, чем у Next.js.

#### Вариант C: Остаёмся на Vite + добавляем react-router
Плюсы:
- Минимальный рефакторинг.
- Не ломаем то что работает.

Минусы:
- Нет SSR → плохой SEO для контент-страниц.
- Нет встроенного MDX → придётся настраивать.
- Для блога / справочника / методологии — без SSR это плохо.

**Рекомендация:** Next.js. Миграция — неделя, но потом каждая следующая
фича дешевле. Особенно блог и каталог видов.

### Auth & data

- **Auth.js / next-auth** с Email-magic-link (без пароля) — самый простой
  UX, провайдеры не нужны.
- **Таблицы пользователя** в том же Postgres:
  ```sql
  CREATE TABLE app_user (id UUID PK, email, created_at, display_name, ...);
  CREATE TABLE user_spot (id, user_id FK, geom GEOMETRY(Point, 4326),
                          name, note, created_at);
  CREATE TABLE user_trip (id, user_id FK, trip_date, species_slug,
                          spot_id FK NULL, note);
  ```
- FastAPI получает новый роутер `/api/cabinet/*` защищённый JWT из
  next-auth.

## 6. Масштабируемость

### Добавление новой вкладки / раздела

С Next.js App Router — тривиально:
```
app/community/page.tsx      → автоматически доступно /community
app/community/[id]/page.tsx → /community/42
```
Плюс навигация обновляется в одном месте (layout.tsx + data/nav.ts).

### Добавление нового типа контента

Каждый новый формат (блог, виды, гайды) = MDX + template. Один раз
настроил layout — все посты/виды наследуют.

### Добавление нового слоя на карте

Паттерн уже описан в CLAUDE.md (миграция → downloader → ingest → build
→ MapView). Он переносится на Next.js как есть — карта живёт в
`app/map/page.tsx`, всё остальное вокруг не меняется.

### Локализация

Next.js i18n встроенный. Сейчас — только ru. Когда-нибудь en/fi —
добавляется без переписывания.

### Производительность

- Статические страницы: Cloudflare CDN → 50ms до пользователя.
- Карта: PMTiles range-requests, lazy-loading слоёв уже работает.
- Lighthouse-цель: 95+ на контент-страницах, 80+ на карте (MapLibre
  тяжёлый сам по себе).

## 7. Мобильный опыт

Уже сейчас базовый адаптив есть (см. коммит «mobile responsiveness»
2026-04-17). Дальше:

### Короткий горизонт
- Проверить на реальном iPhone / Android — не на DevTools Responsive.
- **PWA manifest** — `<link rel="manifest">` + service worker. Делает
  сайт «установимым» на home screen. Дешёвый win, высокая ценность.
- Кнопка **«Моё местоположение»** — `navigator.geolocation` → flyTo.

### Средний горизонт
- **Offline tiles** — загрузить PMTiles для выбранного региона в
  IndexedDB / Cache API. Это значимая фича: в лесу нет интернета.
  Технически реально (PMTiles файл = 496 MB на всю ЛО, выбрать свой
  район и грузить 20-50 MB).
- **Haptic feedback** на клик по полигону — `navigator.vibrate(10)`.
- **Быстрое «сохранить это место»** — одна кнопка, большая, понятная
  жестом.

### Длинный горизонт
- Нативное приложение? Нет смысла если PWA работает. Единственный
  кейс — доступ к камере для фото пойманного гриба в оффлайне. Но
  PWA тоже умеет.

## 8. Деплой — бесплатный

Архитектура нужно балансировать между «карта 500 MB» и «auth + динамический
бэк». Обычный Vercel не подойдёт — serverless-функции не отдадут 500 MB
PMTiles.

### Рекомендуемая связка (бесплатная)

```
┌───────────────────────────────────────────────────────────┐
│  Cloudflare Pages                 (статический Next.js)   │
│  • Пре-билд контент-страниц, SSG                          │
│  • Free tier: 500 builds/mo, unlimited bandwidth          │
└─────┬────────────────────────────────────────────────┬────┘
      │                                                │
      │ /tiles/*.pmtiles                               │ /api/*
      ▼                                                ▼
┌──────────────────┐                       ┌─────────────────────┐
│ Cloudflare R2    │                       │ Oracle Cloud Free   │
│ • PMTiles 500MB  │                       │ • ARM Ampere        │
│ • HTTP Range OK  │                       │ • 4 vCPU / 24GB RAM │
│ • 10GB free      │                       │ • Docker compose:   │
│ • 1M ops/mo free │                       │   FastAPI + PostGIS │
└──────────────────┘                       └─────────────────────┘
```

**Почему именно так:**
- Cloudflare Pages — любой объём трафика бесплатно, уже знает
  Next.js, build-из-git работает коробочно.
- R2 (S3-совместимый) — 10 GB хранения бесплатно, Range-requests
  работают. PMTiles + geojson = ~520 MB, влезаем с запасом.
- Oracle Cloud Free Tier — **единственная** бесплатная VM на которой
  реально запустить PostgreSQL с 2M полигонов. 4 ARM ядра, 24 GB RAM,
  200 GB storage — «всегда бесплатно». Требует карту для регистрации,
  но не списывает (пока в лимитах).

**Альтернативы если Oracle не устраивает:**
- **Fly.io** — 3 shared-cpu-1x с 256 MB RAM бесплатно. Тесновато
  для Postgres + FastAPI, но работает если подтюнить shared_buffers.
- **Railway** — $5 credit/мес, хватит на небольшой API. После —
  платное.
- **Supabase** — Postgres + PostGIS бесплатно, но 500 MB лимит на
  базу. Наши 2M полигонов весят ~3 GB — не влезет. Supabase годится
  для `app_user`, `user_spot`, `user_trip` — маленькие таблицы.

**Гибрид (долгосрочно, когда пойдёт трафик):**
- Пользовательские данные — Supabase free.
- Лесные данные — Oracle Cloud free.
- Два пула, сложнее, но каждый в своей лиге.

### Домен

MVP — `mushroom-map.pages.dev` (Cloudflare даёт субдомен бесплатно).

Платный домен — когда проект вырастет и это станет важно:
- `.ru` — 200–400 ₽/год через reg.ru. Идеи: `mycoforest.ru`,
  `gribnaya-karta.ru`, `gribы.ру` (IDN).
- `.app` / `.com` — $12–15/год через Cloudflare Registrar (продают по
  себестоимости, без markup).

### CI/CD

Уже заложен в `.github/workflows/test.yml`. Расширяем:
1. Push в main → GitHub Actions гоняет тесты.
2. При успехе → Cloudflare Pages автоматически билдит Next.js и деплоит.
3. Oracle VM — `docker compose pull && docker compose up -d` через
   webhook либо вручную.

Это всё бесплатно: GitHub Actions 2000 минут/мес free, Cloudflare deploy
unlimited, Oracle VM — сами управляем.

Каждая фаза — **релизуемый шаг**. Не обязательно делать все подряд,
паузы между ними ок.

### ✅ Фаза 1 — миграция на multi-page Vite (сделано 2026-04-17)
- ✅ `src/routes/` структура, MapView перенесён в `/map`.
- ✅ Главная (`/`) — hero + 3 карточки разделов.
- ✅ `/about` — реальный контент; `/species`, `/guide`, `/methodology`
  — плейсхолдеры (1 общий компонент).
- ✅ SiteHeader + overlay «← На главную» на `/map`.
- ✅ Тот же FastAPI, Vite проксирует `/api/*`.
- ✅ `setStyle({ diff: false })` для стабильного переключения basemap
  после миграции.

**Итог:** `/map` функционально не изменился. Остальные страницы выводят
реальный или placeholder-контент. Билд 1.1 MB bundle — будет фикситься
lazy-load'ом MapView в Фазе 2.

### Фаза 2 — дизайн-система (3–5 дней)
- Tailwind с кастомной темой: цвета, типографика (см. §3 выше).
- Базовые компоненты: Button, Card, NavLink, Tag.
- Переделать `/map` контролы на эту систему (сейчас inline-styles).
- Fraunces + Inter через `@fontsource/*` или CDN.
- Dark mode через CSS-переменные.
- **Lazy-load MapView** (`React.lazy` + `Suspense`) — bundle главной
  должен быть ~50 KB, не 1.1 MB.

**Критерий успеха:** любая новая страница выглядит в едином стиле
без дополнительных усилий; `/` грузится мгновенно, карта подгружается
только при переходе на `/map`.

### Фаза 3 — каталог видов (3–5 дней)
- `/species` — сетка всех 24 видов.
- `/species/[slug]` — детальная страница, генерируется из БД.
- Ссылка с карты-попапа на страницу вида.
- Ссылка со страницы вида «Показать на карте» → `/map?species=<slug>`.

**Критерий успеха:** Google индексирует `/species/*` страницы (проверка
через `site:domain.pages.dev species` через 2–3 недели после деплоя).

### Фаза 4 — деплой на продакшн (1 день + upload)
- Prod Dockerfile для web: `vite build` + nginx-alpine с SPA-fallback.
- Prod docker-compose (api + db): без bind-mount'ов, env через `.env`.
- Oracle Cloud Ampere A1 VM (4 vCPU / 24 GB RAM free) — Ubuntu 22.04,
  Docker, clone repo.
- Caddy как reverse proxy: автоматический Let's Encrypt, маршруты
  `api.domain/*` → FastAPI, `/tiles/*` → R2 proxy (опц.) или напрямую.
- PMTiles + places.geojson → Cloudflare R2 (520 MB). `VITE_API_URL`
  указывает на Oracle VM, но `/tiles/*` — на R2 CDN.
- `pg_dump` локальной БД → upload → `pg_restore` на Oracle.
- Cloudflare Pages project: build `cd services/web && npm ci && npm run build`,
  output `services/web/dist`, env `VITE_API_URL=https://api.<domain>`.
- CORS FastAPI: добавить `*.pages.dev` и будущий домен в `API_CORS_ORIGINS`.

**Критерий успеха:** зайти с телефона через 4G на публичный URL, карта
рисуется, попап работает, страницы видов открываются.

### Фаза 5 — личный кабинет (1 неделя)
- Auth.js с email magic-link.
- Миграции 020+ для app_user / user_spot / user_trip.
- API routes в Next.js (или отдельный router в FastAPI).
- Страница `/cabinet/spots` с CRUD сохранёнными местами.
- Интеграция с картой: «сохранить эту точку» на клике.

**Критерий успеха:** зайти, сохранить место, зайти с другого устройства,
увидеть его на карте.

### Фаза 6 — блог и гайды (непрерывно)

- Первые 3–5 постов — по мере работы над проектом (это и есть контент).
- Лента RSS.

**Критерий успеха:** дубль нет. Это постоянный процесс. Важно просто

### Фаза 7 — PWA + офлайн (3 дня)

## 10. Риски и решения к принятию

### Решения которые надо принять ДО Фазы 1

| Вопрос | Варианты | Моя рекомендация |
|---|---|---|
| Framework | Next.js / Astro / Vite+Router | **Next.js** (универсальнее для auth + контента) |
| UI lib | shadcn/ui / hand-rolled / Mantine | **shadcn/ui** (не lib, а copy-paste компоненты — полный контроль) |
| Tailwind или что-то другое | Tailwind / CSS Modules / Panda | **Tailwind** (экосистема, shadcn-совместимо) |
| Auth | next-auth / Clerk / Supabase Auth | **next-auth** с email magic-link (бесплатно, самохостится) |
| Hosting BE | Oracle Cloud / Fly.io / Railway | **Oracle Cloud Free** (единственный с достаточной памятью для PG+PMTiles) |
| Hosting FE | Cloudflare Pages / Vercel / Netlify | **Cloudflare Pages** (бесплатный трафик) |
| Storage PMTiles | R2 / локально на VM / S3 | **R2** (10 GB free, Range-requests out-of-box) |

### Риски

1. **«Next.js migration — неделя, а может и две»**. SPA → App Router
   не всегда гладко с MapLibre (useEffect-heavy код). Смягчение: делать
   миграцию в отдельной ветке, оставить текущий Vite-билд как fallback
   пока не стабилизируется.

2. **«Oracle Cloud Free Tier как-нибудь отзовут»**. Oracle известен
   агрессивным выключением «always free» аккаунтов с низкой активностью.
   Смягчение: cron-ping раз в неделю чтобы VM считалась «used».
   План Б: Fly.io shared-cpu-1x с подтюненным Postgres.

3. **«PWA-кэш PMTiles в IndexedDB может тормозить»**. 50 MB в
   IndexedDB на мобильном — это граница. Смягчение: кэшировать по
   зумам 10–13, не все; или переключиться на Origin Private File System
   когда Safari добавит.

4. **«Мобильная вёрстка выглядит плохо на реальном телефоне»**. Мы не
   тестировали. Смягчение: в Фазе 2 обязательно открыть на iPhone и
   Android до дизайн-ревизии.
